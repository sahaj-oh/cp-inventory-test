# P4 — Home · OH Properties · Logs · Users · Profile Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax. **Port-and-wire** plan — each task names its source component; complete code only for new glue.

**Goal:** Fill five of the eight staff pages, each porting an existing CP feature into Direct's layout with no functional change (except Home, which is a new summary built from existing data).

**Architecture:** Home is new (built from `adminListSubmissions` stage counts + `ticketsPendingCount`). OH Properties, Logs, and Users port CP's `ExternalInventory`, `ActivityLog`, and `AdminPanel` into Direct's shell. Profile is new-but-thin from `/api/me`.

**Tech Stack:** React 18, react-router, CP `api.js`, `format.js` (all from P2).

## Global Constraints

- **No new endpoints.** Home uses `adminListSubmissions` (stage counts already returned) + `ticketsPendingCount`. OH Properties uses `adminListExternalInventory`. Logs uses `adminListActivityLog` + `adminListActivityLogFacets`. Users uses `adminListStaffUsers`/`adminAddStaffUser`/`adminPatchStaffUser`/`adminForceLogoutUser`/`adminForceLogoutAll`. Profile uses `me()`.
- **Preserve CP semantics:** OH Properties D/F facets + server sort/pagination unchanged; Users' two-table `source` model + admin↔rms boundary guard + viewer-needs-city unchanged; Logs 500-cap + facet filters unchanged.
- **Role gating** already enforced by the P2 router (Logs/Users admin-only; Home/OH Properties/Profile all staff). The **Logs Details renderer must cover `category='ticket'`** rows (spec §10) so nothing renders as raw JSON.

## File Structure
`frontend/src/pages/`: `Home.jsx`, `OhProperties.jsx`, `Logs.jsx`, `Users.jsx`, `Profile.jsx` (all replace P2 stubs).
`frontend/src/components/`: `CardDetailModal` (from P3, reused by Logs), `UserEditModal.jsx` (copy from CP), plus Logs' `Details` renderer inline.
Reference: CP `screens/Admin/{ExternalInventory,ActivityLog,AdminPanel}.jsx`; Direct `pages/{Home,Logs,Users,MyProfile}.jsx`.

## Task 1: Home (summary dashboard)

**Files:** Create `frontend/src/pages/Home.jsx`. Reference: Direct `pages/Home.jsx` (QuadCard/StatTile pattern), CP `Admin/index.jsx` (counts shape).

**Interfaces:** Consumes `api.adminListSubmissions({ limit: 0 })` (or a lightweight call) for `counts`, `api.ticketsPendingCount()`, `useAuth()`.

- [ ] **Step 1: Fetch counts + pending tickets**

```jsx
const [counts, setCounts] = useState({});
const [pending, setPending] = useState(0);
useEffect(() => {
  api.adminListSubmissions({ limit: 1 }).then((d) => setCounts(d.counts || {})).catch(() => {});
  if (user?.role !== 'viewer') api.ticketsPendingCount().then((r) => setPending(r?.count || 0)).catch(() => {});
}, []);
```
(`adminListSubmissions` returns per-stage `counts` regardless of `limit`; a `limit:1` keeps the row payload tiny.)

- [ ] **Step 2: Render the stage summary grid (Direct QuadCard/StatTile)**

Group CP's 9 `STAGES` into cards, each a `<Link to="/submissions?status=…">` with a colored left accent + big Fraunces `StatTile` numbers:
- **Intake** → Unapproved, Submitted
- **Visits** → Visit Requested, Visit Scheduled, Visit Completed
- **Deals** → Offer Given, Closure
- **Rejections** → Price Rejected, Rejected (+ top reasons if present in counts)

Each stat clickable navigates to `/submissions` and pre-selects that stage (the Submissions page reads a `status` query param → `statusFilter`). Reuse Direct's `.home-quad`, `.st-num`, `.stat-tile` classes.

- [ ] **Step 3: Unresolved Tickets card** — a blue-accent count card (`{pending} need your action`) linking to `/tickets`; render only when `role !== 'viewer'`.

- [ ] **Step 4: Build + manual check** → `/` shows summary cards; clicking a stage lands on Submissions filtered. Commit: `feat(home): stage summary dashboard`.

- [ ] **Step 3: Smart `Details` renderer — cover every CP action category (incl. tickets)**

Port CP's `DetailsCell`/`formatDetailValue` and Direct's `<Details>` motif, with explicit branches for CP action types so nothing falls back to raw JSON. **Add ticket branches** (spec §10):

```jsx
function Details({ row }) {
  const d = row.details || {};
  switch (row.action) {
    case 'status_change': return <>Status <b>{d.from}</b> → <b className="val-green">{d.to}</b>{d.status_reason ? ` · ${d.status_reason}` : ''}</>;
    case 'comment': return <>📝 Note by {row.actor_name}: {d.text}</>;
    case 'ticket_created': return <>🎫 Ticket “{d.title}” raised → RM #{d.assigned_rm_id}{d.submission_id ? ` on submission #${d.submission_id}` : ''}</>;
    case 'ticket_reply': return <>💬 Ticket reply: {d.body}</>;
    case 'ticket_closed': return <>✅ Ticket #{d.ticket_id} closed</>;
    // …counter_offer, schedule_visit, bulk_status, staff mutations, sync — port CP's existing branches…
    default: return <KeyValues d={d} />;   // up-to-5 key:value pairs, "+N more"
  }
}
```
`UID` cells starting with `OHL` render as an orange link → open `CardDetailModal` (P3) for that submission by public_id.

- [ ] **Step 4: Build + manual check** → `/logs` renders CP's log with Direct styling; ticket rows show friendly summaries, not JSON; UID opens the property modal. Commit: `feat(logs): activity log in Direct shell + ticket-aware details`.
